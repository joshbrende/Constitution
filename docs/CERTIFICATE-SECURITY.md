# Certificate Security Runbook

This runbook defines the security posture, verification workflow, revocation procedure, and incident response playbook for certificate issuance and verification.

---

## 1) Scope and objectives

### In scope
- Membership certificates generated by backend services.
- Public certificate verification endpoint and QR validation.
- Admin revocation/reinstatement operations.
- Data fields used for integrity and lifecycle status:
  - `certificate_number`
  - `verification_code`
  - `public_id`
  - `expires_at`
  - `revoked_at`

### Security objectives
- Prevent easy forgery and enumeration.
- Ensure verifiers can confirm authenticity and status.
- Enable rapid revocation when compromise is suspected.
- Provide repeatable evidence for stakeholder/security audits.

---

## 2) System summary

### Generation
- Certificates are issued on pass of membership assessment.
- PDF generation includes verification QR linking to public verify endpoint.
- QR includes verification attributes: `id`, `number`, `code`, `token`.

### Verification
- Public endpoint: `GET /verify-certificate`
- Required inputs for successful verification:
  - Certificate number (or public ID)
  - Verification code
- Optional signed token is validated when present.
- Verification status shown to verifier:
  - `active`
  - `expired`
  - `revoked`

### Operations
- Admin can revoke and reinstate certificates from admin certificates view.
- Default expiry policy can be configured via:
  - `CERTIFICATE_DEFAULT_EXPIRY_DAYS` (default `730`)

---

## 3) Threat model

### Assets
- Certificate authenticity and trust.
- Verification service availability.
- Issuance metadata and lifecycle status.
- Signing secret (`APP_KEY` used by HMAC validation).

### Adversaries
- Opportunistic fraudsters modifying PDFs.
- Automated bots attempting brute-force verification.
- Insider misuse of revocation controls.
- Attackers with leaked QR links attempting replay/tamper.

### Threats and controls

- **T1: Enumeration of certificate IDs**
  - Risk: attacker guesses valid certificates.
  - Controls:
    - Non-sequential certificate numbers.
    - UUID `public_id`.
    - Verification code required.

- **T2: Verification brute force**
  - Risk: attacker repeatedly submits combinations.
  - Controls:
    - Route throttling on `/verify-certificate`.
    - Combined limits by IP and IP+identifier tuple.

- **T3: QR parameter tampering**
  - Risk: modified URL still appears valid.
  - Controls:
    - HMAC signed verification token.
    - Server-side token validation (when token present).

- **T4: Use of revoked/expired certificate**
  - Risk: stale or compromised certificates accepted.
  - Controls:
    - Lifecycle fields `revoked_at` and `expires_at`.
    - Verification status rendered as active/expired/revoked.

- **T5: Operational delay during compromise**
  - Risk: compromised certificate remains active too long.
  - Controls:
    - Immediate admin revoke action.
    - Incident response SOP in this runbook.

Residual risk: if application key is compromised, token trust degrades; rotate key/secrets and reissue where necessary.

---

## 4) Verification flow (auditable)

1. Verifier scans QR or opens verification page.
2. Request hits `GET /verify-certificate` (throttled).
3. Backend requires verification code for success.
4. Backend resolves certificate by:
   - `public_id` when supplied, otherwise `certificate_number`
   - plus exact `verification_code`
5. If token is supplied, backend validates HMAC token.
6. Backend computes status:
   - `revoked` if `revoked_at` is set
   - `expired` if `expires_at < now`
   - otherwise `active`
7. UI displays recipient/course/date and verification status.

### Success criteria
- Only matching code + identifier returns a record.
- Status is visible and correct.
- Token mismatch fails verification path.

## 5) Revocation SOP (standard operating procedure)

### Triggers for revocation
- Reported fraud or altered certificate document.
- Issuance error requiring invalidation.
- Identity mismatch discovered post-issuance.
- Security incident involving certificate metadata integrity.

### Revoke steps (admin)
1. Open **Admin -> Certificates**.
2. Locate certificate by number / verification code / member name.
3. Confirm record details (recipient, course, issue date).
4. Click **Revoke** and confirm prompt.
5. Verify status changes to `revoked`.
6. Perform external verification check on public endpoint.
7. Record case in audit register (reason, actor, timestamp).

### Reinstate steps (admin)
1. Locate revoked certificate.
2. Validate supporting approval/evidence.
3. Click **Reinstate**.
4. Confirm status returns to `active` (or `expired` if past expiry).
5. Update audit register with approver and rationale.

### Controls
- Revoke/reinstate is restricted to admin-protected routes.
- Actions should be tied to authenticated admin sessions.

## 6) Incident response playbook

### Severity guide
- **SEV-1**: Active widespread forgery or key compromise.
- **SEV-2**: Single-certificate compromise with public impact.
- **SEV-3**: Suspicious activity without confirmed abuse.

### Immediate actions
1. Triage report and collect indicators:
   - certificate number / public ID
   - verification URL
   - screenshots/documents
   - timestamps and source channel
2. Revoke affected certificate(s) immediately.
3. Validate public verification now reports `revoked`.
4. Notify stakeholders:
   - product owner
   - security lead
   - support team

### If key compromise suspected
1. Rotate application signing secret according to platform policy.
2. Invalidate trust in prior signed tokens.
3. Reissue affected certificates where required.
4. Publish communication for external verifiers if broad impact.

### Containment and recovery
- Increase verification throttling temporarily if under attack.
- Monitor verification endpoint rates and invalid attempts.
- Run targeted data checks for suspicious issuance patterns.

### Post-incident
- Complete root-cause analysis within agreed SLA.
- Document timeline and corrective actions.
- Add preventive controls/tests before closure.

## 7) Audit evidence checklist

For stakeholder/security audit packets, include:
- Current schema evidence for:
  - `public_id`, `verification_code`, `expires_at`, `revoked_at`
- Verification route config showing throttle middleware.
- Sample verification screenshots:
  - active
  - expired
  - revoked
- Admin operation evidence:
  - revoke and reinstate workflow
- Config snapshot:
  - `CERTIFICATE_DEFAULT_EXPIRY_DAYS`
- Change log references in `docs/progress.md`.

---

## 8) Operational checks (monthly)

- Randomly sample active certificates and verify status correctness.
- Sample revoked certificates and confirm they remain non-active.
- Confirm expiry transitions are reflected as expected.
- Review verification endpoint logs for brute-force patterns.
- Validate admin access controls remain restricted.
