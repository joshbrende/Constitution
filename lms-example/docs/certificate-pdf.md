# Certificate PDF

## Overview

Learners can download their certificate as a PDF from the certificate view. The system supports two modes:

1. **Background PDF mode** (when `public/asset/certificate_background.pdf` exists and FPDI/TCPDF are installed): The certificate is generated by overlaying dynamic text and signature images on the background PDF. Student **name and surname** appear on the certificate; the **certificate number** is system-generated.
2. **Fallback** (Dompdf): A simple HTML-based PDF when the background PDF or packages are not available.

---

## Prerequisites

- **Dompdf** (fallback): `composer require barryvdh/laravel-dompdf`
- **Background PDF mode**: `composer require setasign/fpdi tecnickcom/tcpdf`
- Place **certificate_background.pdf** in `public/asset/certificate_background.pdf`.
- **Great Vibes font** (optional): Place `GreatVibes-Regular.ttf` in `storage/app/fonts/GreatVibes-Regular.ttf` for script-style text. If missing, the service uses a built-in serif font.
- Ensure `storage/app/fonts/` exists; TCPDF may need a writable font cache (e.g. `vendor/tecnickcom/tcpdf/fonts` or configure `K_PATH_FONTS`).

---

## Signatures

Three signature images can appear on certificates:

| Role | Who uploads | Where |
|------|-------------|--------|
| **Board of Faculty** | Admin | Admin → Certificate signatures |
| **Supervisor** | Admin | Admin → Certificate signatures |
| **Facilitator** | Facilitator (per user) | Instructor → Certificate signature |

Signatures are stored in `storage/app/certificate/signatures/`. Admin uploads one image each for Board of Faculty and Supervisor. Each facilitator uploads their own image; it appears on certificates for courses they instruct (based on `course.instructor_id`).

## Routes

| Method | URI | Name | Purpose |
|--------|-----|------|---------|
| GET | `/certificates/{certificate}` | `certificates.show` | View certificate (own only) |
| GET | `/certificates/{certificate}/pdf` | `certificates.download-pdf` | Download PDF (own only) |
| GET | `/certificates/preview-pdf` | `certificates.preview-pdf` | Preview PDF (facilitator/admin) |
| GET | `/admin/certificate-signatures` | `admin.certificate-signatures.index` | Admin: upload Board of Faculty & Supervisor |
| POST | `/admin/certificate-signatures` | `admin.certificate-signatures.store` | Admin: store signature |
| GET | `/instructor/certificate-signature` | `instructor.certificate-signature` | Facilitator: upload own signature |
| POST | `/instructor/certificate-signature` | `instructor.certificate-signature.store` | Facilitator: store signature |

---

## User flow

1. **Admin**: Upload Board of Faculty and Supervisor signatures (Admin → Certificate signatures).
2. **Facilitator**: Upload own signature (Instructor → Certificate signature).
3. **Learner**: Complete a course → certificate is created with system-generated certificate number. Name and surname come from the user record.
4. **Learner**: My courses → course → View certificate → **Download PDF**.
5. PDF uses the background template (if present), overlays name, surname, course title, certificate number, date, and the three signature images where configured.

## PDF content (background mode)

- Title: “Certificate of Completion”
- “Is hereby awarded to”
- **Recipient name and surname** (from user)
- “As recognition for the participation in the training programme entitled”
- **Course title**
- **TTM Group**
- **Certificate no.** {system-generated}
- **Date** (issued_at)
- Signature images: Facilitator (left), Board of Faculty (centre), Supervisor (right) when uploaded.

---

## Access

- Learners can only view and download their own certificates (`user_id` match).
- Preview is restricted to users with `canEditCourses()` (facilitators and admins).
- Certificate signatures: Admin for Board of Faculty and Supervisor; facilitators for their own signature.
